## 4. Supabase RPCs and Edge Functions

The legacy frontend uses a hand-rolled HTTP client (`supabaseRpc(fn, payload)`) that posts to `/rest/v1/rpc/${fn}`. The new app replaces this with `@supabase/supabase-js`.

### RPCs (exact names as called in code)

| RPC name | JS wrapper | Views / contexts |
|----------|-----------|-----------------|
| `get_dashboard_state` | `remoteLogin` / `loadSharedStore` | Login gate; returns shared state (targets, likelihood flags, settings) consumed by all views |
| `save_dashboard_state` | `pushSharedStoreNow` | Any view that mutates help-needed flags, operating manual inputs, or targets |
| `get_dashboard_version` | `fetchVersionData` / `fetchVersionDataCached` | **All data views** — Scorecard, Stage Conversion, Compare, Momentum, LT Trends, Operating, Vertical Performance (v1+v2), Funnel, Industry, Win/Loss, Intro Trend, Cycle Time, Forecast, Forecast Actuals, Admin |
| `get_dashboard_version_qa` | `fetchVersionQa` / `fetchVersionQaCached` | Admin (data quality panel) |
| `get_dashboard_compare` | `fetchVersionCompare` | Version Compare, Momentum |
| `rename_dashboard_version` | `renameVersion` | Admin |
| `upsert_dashboard_user` | `upsertUser` | Admin (create user) |
| `delete_dashboard_user` | `deleteDashboardUser` | Admin (delete user) |

**Note:** There is no separate `verify_dashboard_user` RPC. Authentication reuses `get_dashboard_state` with username+password params; a successful return = authenticated.

## 5. Broken patterns and flags

### 5.1 `view-leadershipforecast-v2` fork (critical — drives Phase 2)

Two view divs for the same conceptual screen coexist:

- `view-leadershipforecast` (main nav): v1 table layout — **was** the primary "Vertical Performance" view
- `view-leadershipforecast-v2` (Appendix): v2 executive redesign — **demoted to Appendix in v2.1.4** after being the primary view in v2.1.0–v2.1.3

Version history is explicit: `v2.1.4` "Restored the main Vertical Performance screen to the v2.0.15 table layout and moved the redesigned executive-clean version into Appendix as Vertical Performance v2." This was a Codex regression under pressure — the v2 design is materially better and should be the only view. Phase 2 implements it as the canonical `VerticalPerformance.tsx` and retires v1 entirely.

### 5.2 Hardcoded Supabase credentials (security)

The anon key (a long JWT) and project URL are string literals in the script. The anon key is public by Supabase design but must not be in source. Move to env vars before first deploy.

### 5.3 Session password persisted in localStorage (security concern)

`saveSession()` writes the user's cleartext password to `localStorage` (key `sales_dashboard_session_v1`) for auto-login on next visit. In the new app, use Supabase Auth sessions or at minimum `sessionStorage` (cleared on tab close). Flag for Phase 3.

### 5.4 Hand-rolled Supabase HTTP client

`supabaseRpc` manually constructs `fetch` calls against the Supabase REST endpoint (no SDK). This loses: auto-retry, typed responses, RLS error propagation, realtime. Replace with `@supabase/supabase-js` in Phase 1.

### 5.5 Stage numbering inconsistency

`MATTER_STAGE_ORDER` and `FUNNEL_ORDER` both list `6. Commercial Proposal` before `5. Contracting` — the board numbering disagrees with logical pipeline order. This is intentional (Monday.com column numbering artifact) but causes reader confusion. In `src/lib/stages.ts`, define one canonical array with a comment explaining the 5/6 inversion.

### 5.6 Crosstab backed by static JSON, not Supabase

`view-crosstab` loads `crosstab_data.json` generated offline by `build_crosstab_data.py`. The file is in the repo root. In the new app: move to `public/crosstab_data.json`, fetch client-side, and label the view clearly as "static data." Connecting it to a live RPC is Phase 3+ work.

### 5.7 Closure Priorities inputs are localStorage-only

All deal scoring inputs (need strength, lever gaps, recommended actions) are stored only in the user's browser. The footnote acknowledges this. Flag for Phase 3 backend persistence via `save_dashboard_state`.

### 5.8 Dangling `var(--text)` at line 2320

`color: var(--text)` with no `:root { --text: ... }` definition. Renders as inherited color (black). Replace with `var(--text-primary)` in the migrated component.

### 5.9 30+ module-level `let` variables for view state

All per-view state (`scorecardSelectedSeller`, `leadershipForecastSelectedSeller`, `operatingEditMode`, etc.) are module-level `let` variables, creating implicit cross-view coupling. React component state + URL search params (`useSearchParams`) replace all of these.

### 5.10 Two separate EV probability sets (inconsistency)

- `view-forecast` footnote: Intro 5%, Qual 10%, Cap 20%, Scoping 50%, Proposal 70%, Contracting 80%
- `view-leadershipforecast-v2` footnote and `FORECAST_STAGE_CONFIG`: Intro 8%, Qual 10%, Cap 19%, Problem Scoping 29%, Proposal 44%, Contracting 90%

The v2 set ("empirical EV") is the current business rule. The v1 set in `view-forecast` is legacy and should be reconciled. Confirm with user which probabilities `Forecast.tsx` should use in Phase 3.

### 5.11 No Codex artifacts found

No `.codex/` directory or `AGENTS.md` present. `MIGRATION_PROMPT.md` at repo root is the previous agent's handover document — preserve as-is.
